# DMAD 오탐 분석

> 이 문서는 예비 실험에서 발견한 오탐 메커니즘과 차단 방법을 기록한다.

## 발견 경위

A/B 테스트 중 순차 DMAD에서 오탐이 발생했다.
T1-B1 (Sonnet + 풀버전 + 순차): content_type 백슬래시 주장 → 실제 코드는 슬래시

## 오탐 유형 2가지

### 1. 코드 오독 할루시네이션
파일을 실제로 읽지 않고 기억으로 코드를 설명하다가 틀림.

**예시:**
- 설계자: "audio/{ext}가 백슬래시로 되어있어서 문제"
- 실제 코드: `f"audio/{file_ext[1:]}"` (슬래시)

**차단 방법:** 토론 시작 전 파일 Read 강제 + 코드 인용 규칙

### 2. 도메인 지식 부재
코드를 정확히 읽었지만 실행 컨텍스트를 모름.

**예시:**
- "ffprobe가 MP4 이후에 실행되는 순서가 문제"
- 실제로는 로컬 다운로드 후 실행이라 문제없음

**차단 방법:** 없음. 실행만이 해결. CoVe 검증으로 일부 필터링 가능.

## 할루시네이션 전파 메커니즘

순차 DMAD에서 발생하는 패턴:

```
설계자 오판 (잘못된 코드 설명)
    ↓
사용자가 설계자 설명을 믿고 같은 방향으로 재확인
    ↓
오탐 생존 (두 역할 모두 동의)
```

병렬에서 차단되는 이유:
```
설계자 독립 분석 (파일 직접 읽기)
사용자 독립 분석 (파일 직접 읽기)
    ↓
설계자 오판이 사용자에게 전파되지 않음
    ↓
사용자가 실제 코드 읽고 "슬래시 맞는데?" → 오탐 교정
```

## 적용된 규칙

```
[강제] DMAD 시작 전 변경 대상 파일 + 직접 의존 파일 전부 Read
[필수] 문제 제기 시 파일명:라인: 코드내용 형식으로 직접 인용
       인용 없는 주장 금지
```
